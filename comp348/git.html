Point of Clarification

Git is a version control system. It is used to version-control source code. GitHub/GitLab/Bitbucket/(insert provider name here) is a Git provider; that is, they provide Git services. They are not the same thing!

Git, Conceptually

Rather than save your entire file every single time (expensive and timely), Git finds the differences (the diff) between your old code and your new code, then saves those differences.

This makes committing files and finding changes much faster.

Repositories

A “source of truth” codebase must exist for a project. This is the code that everyone will compare theirs to and merge into.

This is know as the remote repository, and only one copy of the remote should exist.

When you make a copy of the remote repository onto your local machine, you are making a local repository. Making a local repository is called cloning the repository. Many local repositories can exist simultaneously.

Change Logging

Many times, you don't want to sync your changes with the remote repository immediately. You might need to run unit tests or debug your code before you push it to the remote repository. When you're ready to, you will stage and commit those changes.

  • Staging a change tells Git that you're ready to commit those changes to the remote repository, but it doesn't actually make the change quite yet.
  • Committing a change tells Git to put the staged change into the permanent ledger of the repository and to change the codebase to reflect the staged changes.

Syncing

When you stage and commit a change, it is not immediately synced. Instead, a pull and push process must be run.

  • The pull process gets the latest changes from the remote codebase. This makes sure that any changes that were made before you sync are reflected in the new codebase.
  • The push process takes your changes, and as its name implies, pushes them to the remote repository.

Cloning

When a project begins, it only exists in the remote repository, like the copy in GitLab. In order to make changes to the repository, you need to make a local copy of that repository that you can edit.

The most common clone method used today is HTTPS.

The .gitignore File

The .gitignore file is a normally hidden file. On macOS, press Cmd+Shift+. to show hidden files.

Use the .gitignore file to tell Git what files and directories you always want to ignore (not commit).

A collection of good .gitignore files can be found at https://github.com/github/gitignore.

Asterisks * are used as the wildcard character, and pound signs # are comments.

Make a Change

Change the files in your local repository.

Stage

Stage your changes — add your changes to the next commit. Any changes that aren't staged won't be added.

Commit

Commit your changes — the staged changes are marked to be added to the codebase on the next push.

If you pass the -m option and a string when making a commit, Git will use that string as the commit message.

If you only use git commit, Git will open a text editor in the terminal for you to write a commit message. Git will typically use Vi unless told otherwise.

Bare essentials Vi: Press i to enter Insert mode, then use the arrow keys to navigate. To save, press Esc, then : (colon), then w (for write), then q (for quit), then Enter.

Alternatively, just skip all of this and use the -m option when committing.

Good commit messages...

  • Are short (<72 characters)
  • Are descriptive
  • Are succinct

Instead of "I added a print line to the output function on line 73 of the main file," say "Add print to output function".

Remember, Git is recording the committer, file, line number, and change for you. You're only adding the human-readable notes.

While Squashed some bugs and More stuff are both short, they're not very descriptive. Add name, age, height, weight, and salary labels with corresponding text boxes and more over a submit button is too long and is not very succinct. Fix null pointer exception in user login hits a good balance of description and brevity. If someone wants to see exactly what was changed, they can look directly at the code.

Pull and Push

Before you make your changes to the remote repository, first pull. This gets the latest version of the remote repository into your local repository. This will help stop merge conflicts.

After you've synced the remote repository with your local repository, push your code changes into the remote repository.

Branches

Branches allow different people to work on different components of a piece of software without worrying that their branch will affect the main trunk. Branches can be used to develop, test, and validate software before it is put into production.

There are two ways to create a branch: using the web GUI in GitLab, then pulling it down to your local repository, or by using the command line interface and pushing it up to the remote repository.

Neither method is better or worse than the other; it's all about what works for you.

If the branch already exists, you can simply switch to it.

In GitLab, click on the branches list, then select the branch you want to view. This only affects what you see in the web interface, not what's on your local machine.

In the command line interface, use the checkout command.

Do not use the -b option — this will create a new branch with a duplicate name, which is a bad thing!

Merging

Merging is the process of taking the changes in a branch and combining them with another branch (often the main or master branch).

The preferred method to merge your branches is by using the GitLab web interface. Some extra checks are made before the merg is actually executed to make sure that merge conflicts won't occur or can be addressed earlier. Forcing a merge using the CLI is not recommended, since these checks are not made until execution time and changes actually start being made.

In the perfect world, there wouldn't be anything wrong with merging in the CLI, but since merge conflicts can occur fairly easily in larger development projects, it's important to carry out these checks beforehand.
